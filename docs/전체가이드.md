# GUIDE — 전체 가이드 (중앙 모달 + 실제 페이지 iframe 미리보기 + 하이라이트 오버레이)

## 개요
홈 화면 상단의 "전체 가이드" 버튼을 클릭하면, 화면 정중앙에 모달이 뜨고 실제 업무 흐름 순서대로 각 페이지를 소개한다. 서비스의 본질이 "요청서를 작성하고 결재를 진행하는" 전자결재형 워크플로우이므로, 가이드도 그 흐름을 그대로 따라간다.

**단계 순서 (내러티브 기반):**
1. **결재 경로** — 최종 결재 경로를 가장 먼저 개관 (독립 단계, iframe 아님)
2. 요청서 작성
3. 결재 진행(결재 현황)
4. 이력 조회 *(미구현)*
5. VOC *(미구현)*
6. 권한관리

각 단계는 정적 스크린샷이 아니라, **실제 라우트를 iframe으로 그대로 임베드**하여 현재 페이지 모습을 그대로 보여주고, 그 위에 **하이라이트 박스 + 가짜 커서 + 설명 캡션을 순차로 얹는 오버레이**로 핵심 요소를 안내한다. 단, **결재 경로·권한관리 단계는 iframe이 아니라 컴포넌트형**으로 렌더한다(아래 참고). 모달은 항상 화면 중앙에만 표시하고, **상단에 페이지별 탭**을 두어 아무 단계로나 자유 이동할 수 있다.

> ⚠️ **이 가이드는 기존 가이드 시스템과 별개로 새로 만드는 기능이다.**
> 현재 프로젝트에는 이미 (1) `/guide` 메뉴의 **기능별 가이드 지식베이스**(`pages/GuidePage.tsx`), (2) 우측 **슬라이드 패널**(`components/GuideSlidePanel.tsx`, framer-motion), (3) 단계별 **애니메이션 데모 컴포넌트**(`components/guideDemos/` — `Step1LineProcessDemo` 등 9개 + `GUIDE_DEMOS` 매핑)가 존재한다. 본 "전체 가이드"는 이들과 **목적·진입점·표시 방식이 다른 별도 기능**이며, 기존 `/guide` 메뉴를 대체하지 않는다. (단, 애니메이션 루프 구현 패턴/유틸은 기존 `guideDemos`를 참고·재사용할 수 있다 — "향후 확장" 참고)

프로토타입 참고: `docs/animated_demo_guide_mockup.html` (정적 HTML/CSS/JS, 단계별 애니메이션 루프 및 취소 로직 포함)

## 트리거
- **홈 화면 Hero 영역의 "전체 가이드" 버튼 클릭 (유일한 진입점)** — 이 버튼은 **신규로 추가해야 한다.** `pages/HomePage.tsx`의 Hero 섹션(의뢰서 작성 / 결재 현황 링크 버튼 영역)에 함께 배치한다. 기존 `/guide` 메뉴 링크(기능별 가이드 지식베이스)와는 별개 기능이다.
- 가이드는 **버튼을 눌렀을 때만 표시**된다. 자동 팝업/로그인 시 자동 시작은 없으며, 따라서 "다시 보지 않기" 및 노출 여부 영속성 저장도 두지 않는다.

## 데모 구현 방식 (결정됨 — iframe 임베드 + 오버레이)
"현재 페이지 모습을 그대로" 보여주기 위해, **실제 라우트를 iframe으로 임베드**한다.
- **임베드 모드**: `App.tsx`에서 라우트 쿼리 `?embed=tour`를 감지하면 `Navbar`/`Footer`/세션 경고 모달 없이 **페이지 본문만** 렌더(`.app-tour-embed`). iframe `src`는 `'/{path}?embed=tour'`.
- iframe은 **같은 오리진**이므로 `contentDocument`에 접근 가능 → 실제 페이지 요소를 **선택자로 조회**해 위치를 읽고 그 위에 하이라이트/커서/캡션을 그린다. (실제 코드·실데이터 그대로, 디자인 변경 시 자동 동기화)
- 가상 뷰포트(1180×720)로 렌더한 뒤 모달 너비에 맞게 **CSS `transform: scale()`로 축소**. 요소 좌표는 `getBoundingClientRect() × scale`로 매핑.
- iframe은 `pointer-events: none`(쇼케이스 전용, 사용자가 내부 조작 불가). 가이드 네비게이션은 모달이 제어.
- **일시정지**: 오버레이 진행(다음 phase로 이동)을 멈춘다.

### 요청서 작성 — 위저드 투어 모드 (RequestPage 협조)
요청서 위저드의 2~5단계(MAP/J-ayer/O-ayer/BB) 화면을 "값이 채워진" 상태로 보여주기 위해 `RequestPage`에 **투어 모드**를 추가했다 (`?embed=tour` 가드, 평상시 동작 무영향).
- `?embed=tour` 진입 시 `detail`/`jayerRows`/`oayerRows`/`bbRows` 초기 상태를 **샘플 값으로 시드**(`constants.ts`의 `makeTour*`). `isLoadingEditRef`를 `true`로 시작해, 라인/조합법 변경 시 하위 필드를 비우는 reset 효과가 시드 값을 지우지 않게 한다.
- 위저드 단계 전환은 **`postMessage`**(`{type:'guide-tour-step', step}`)로 처리 — iframe을 매번 리로드하지 않아 깜빡임이 없다. `RequestPage`는 투어 모드에서 이 메시지를 받아 `setStep`한다.
- 선택자가 없는 요소엔 `data-tour` 속성을 추가:
  - 위저드 인트로: `WizardIndicator`의 각 단계 노드 `.wizard-step[data-step="N"]`
  - MAP(StepMap): `[data-tour="map-purpose"]`(목적) / `[data-tour="map-cfamily"]`(C가문) / `[data-tour="map-deviation"]`(지도편차) / `[data-tour="map-exception"]`(예외구역) / `[data-tour="map-xmark"]`(X표시)
  - J-ayer(Step2): 셀 위치 식별용 `data-jtour="product_name|step|item_id-{행idx}"`, 필터 `[data-tour="jayer-filter"]`
  - O-ayer(Step3): 탭 `[data-tour="oayer-tabs"]`, 정보 탭 TBV/TLV `[data-tour="oayer-info-tbvtlv"]`
  - BB(Step4): 자동채움 `[data-tour="bb-autofill"]`, 분할 패널 `.bb-split-panel`
  - BB 매핑 커서 데모용 `data-bbtour`: 외부 데이터 탭 `bbtab-{idx}`, 원본 행 `jrow-{layerid}`, 외부 행 `ext-{layerid}`, 매핑 적용 `map-apply`
  - 상신 모달: 특이사항+검토자 묶음 `[data-tour="submit-fields"]`

리스크: 첫 iframe 로드 약간의 지연 · phase 선택자가 실제 마크업에 의존(마크업 변경 시 갱신 필요) · 비동기로 늦게 렌더되는 요소는 폴링(최대 2.6s)으로 대기.

## 컴포넌트 구조 (React / TypeScript)
- Hero 버튼: `pages/HomePage.tsx` Hero 섹션에 "전체 가이드" 버튼 → 클릭 시 `GuideTourModal` 오픈
- `App.tsx` — `?embed=tour` 임베드 모드 분기 (네비/푸터/세션 모달 없이 페이지 본문만 렌더)
- `components/GuideTourModal.tsx` — dim 오버레이 + 중앙 모달, step 상태(`current`)·일시정지 상태(`paused`) 관리, `useGuideTourSteps()` 포함
- `components/GuideTourStepPreview.tsx` — `step.path`를 iframe으로 띄우고 `step.phases`를 순차 진행하는 오버레이 엔진(축소 비율 계산, `postMessage`로 위저드 단계 전환, iframe DOM 폴링/조회, `active`/`paused`/취소 반영)
- `pages/RequestPage/index.tsx` — 투어 모드(시드 + `guide-tour-step` 메시지 수신)
- 단계별 전용 데모 컴포넌트는 **두지 않는다** — 각 단계는 데이터(`path` + `phases`)로만 기술되고, 렌더링은 `GuideTourStepPreview`가 공통 처리한다.

## 데이터 모델
```ts
interface TourPhase {
  wizardStep?: number; // /request 위저드 단계(1~5). 지정 시 해당 단계로 전환 후 강조
  selector: string;    // iframe 내부 페이지 요소 선택자
  captionKey: string;  // 설명 캡션 i18n 키
}
interface GuideTourStep {
  key: 'request' | 'approval' | 'history' | 'voc' | 'permission';
  title: string;
  description: string;
  path: string;        // iframe으로 띄울 실제 라우트 (예: '/request')
  phases: TourPhase[]; // 한 단계 안에서 순차로 진행할 phase들
}
```
- `GuideTourStepPreview`는 `active === true`일 때만 오버레이 루프를 실행하고, step이 바뀌면(컴포넌트 key 변경) 정리된다.
- `paused === true`이면 다음 phase로 진행하지 않고 현재 강조를 유지하며, `false`로 돌아오면 이어서 진행한다.

**step key ↔ 실제 라우트 매핑** (`components/Navbar.tsx`의 `NAV_LINKS` 기준):

| key | 라우트 | 페이지 컴포넌트 |
|---|---|---|
| `request` | `/request` | `pages/RequestPage/index.tsx` |
| `approval` | `/approval` | `pages/ApprovalPage.tsx` |
| `history` | `/history` | `pages/HistoryPage.tsx` |
| `voc` | `/voc` | `pages/VOCPage.tsx` |
| `permission` | `/permissions` (복수형 주의) | `pages/PermissionPage.tsx` — 단, 투어는 iframe이 아니라 `PermissionUserGroupDemo` 컴포넌트를 임베드 |

> "시작하기"로 가이드를 종료할 때 특정 페이지로 이동시키려면 위 라우트를 사용한다. 특히 권한관리는 경로가 `/permissions`(복수)임에 유의. (권한관리 단계는 iframe 대신 컴포넌트형 데모를 사용 — "권한관리 — 컴포넌트형 단계" 참고)

## 오버레이 엔진 구현 가이드 (`GuideTourStepPreview`)
- iframe은 첫 phase의 `wizardStep`으로 **한 번만 로드**(`?embed=tour&step=N`). `onLoad` 후 루프 시작.
- 루프: 각 `phase`마다 → 필요 시 `postMessage`로 위저드 단계 전환(`cmd:'step'`) → `phase.cmd`가 있으면 데모 명령 전송 → 선택자 요소 폴링(최대 2.6s) → `scrollIntoView` 후 `getBoundingClientRect() × scale`로 강조/커서/캡션 배치 → `phase.hold` 대기 → 다음. 선택자 없는 phase는 캡션을 중앙 표시. 모든 phase 후 재시작.
- **축소/높이**: 너비뿐 아니라 `window.innerHeight × 0.52`로도 제한해 컨트롤 버튼이 한 화면에 보이도록 함.
- **일시정지/시킹**: `wait(ms)`는 취소·일시정지·seek를 반영. 부모(모달)가 `seek={index,nonce}`를 바꾸면 다음 반복에서 해당 phase로 점프하며 `onPhaseChange(i)`로 현재 phase를 모달에 알린다.
- **되감기 정확 복원(요청서 단계 한정, 스냅샷/복원 방식)**: 정주행 중 각 phase가 정착(hold 완료)되면 프리뷰가 `snapshot` 명령으로 iframe의 현재 투어 상태(step·detail·jayer/bb rows·패널·모달·탭 등)를 회신받아 `snapshotsRef[챕터]`에 캐시한다(`guide-tour-state` 메시지). 타임라인에서 챕터를 클릭(seek)하면, 그 챕터의 스냅샷이 있으면 `restore` 명령으로 **즉시 한 번에 그 시점 상태를 주입**하고 cmd 재생은 건너뛴다 → 깜빡임·지연·토스트 없이 정확 복원(예: '상신하기'→'bb매핑' 되감기 시 상신 모달이 정확히 사라지고, 'bb매핑'→'자동채움' 되감기 시 BB제품2 행이 사라진 시점으로 복원). 아직 정주행으로 도달하지 않아 스냅샷이 없는 챕터로 **앞으로 점프**하면 기존처럼 해당 phase로 직접 점프(폴백). 결재현황 등 다른 단계도 직접 점프(스냅샷 미사용).
- **단계 이동 초기화**: 모달의 `이전/다음`·상단 탭 이동(`goPrev/goNext/goTo`)은 공통 `resetPlayback`으로 phase·seek를 0으로 초기화 → 다음 가이드가 항상 첫 챕터부터 시작.
- iframe DOM 접근은 동일 오리진 전제. 접근 실패 시 `try/catch`로 무시.

### 부모 → iframe 명령 프로토콜 (`postMessage`, `RequestPage` 투어 모드가 수신)
| cmd | 동작 |
|---|---|
| `step` (+`step`) | 위저드 단계 이동(+ 상신 모달/패널 닫기, 투어 커서/칩 정리) |
| `snapshot` (+`index`) | 현재 투어 상태를 부모로 회신(`window.parent`에 `{type:'guide-tour-state', index, state}`). 진행 중 데모를 취소하지 않는 읽기 명령 |
| `restore` (+`state`) | 부모가 캐시한 스냅샷(`TourSnapshot`)을 받아 그 챕터의 상태를 즉시 복원(`applySnapshot`) + 커서/칩 오버레이 정리 |
| `map-reset` | MAP 필드를 시드 상태로 초기화(목적 NEW, 나머지 변경 없음) |
| ~~`map-cfamily`~~ | (제거됨) C가문 토글 연출은 삭제 — 이제 `map-cfamily` phase는 필드(`[data-tour="map-cfamily"]`)를 강조·설명만 하고 값은 바꾸지 않는다 |
| `map-deviation` | 지도 편차 `변경 있음` + X/Y·사유 샘플 입력 |
| `map-exception` | 예외 구역 `변경 있음` + 예외 구역 값 샘플 입력 |
| `map-xmark` | X표시 변경 `추가` → 이미지 첨부 영역 노출(실제 사진 붙여넣기 안내) |
| `jayer-anim` | **실제 J-ayer 표 위**에서 가짜 커서 이동 + 셀 선택 + Ctrl C/V 칩으로 1행 입력→복사→드래그 붙여넣기→`step`·`item_id`(barcode) 자동 채움 연출 |
| `oayer-table` / `oayer-info` | O-ayer 탭을 표/정보로 전환 |
| `bb-autofill-open` | 자동채움 패널 열기만(설명용) — 적용은 누르지 않는다 |
| `bb-autofill-apply` | **커서를 '적용' 버튼(`[data-bbtour="autofill-apply"]`)으로 이동(스크롤로 보이게) → 눌러서** BB제품1 3행 생성 |
| `bb-mapping` | (실제 핸들러) **가짜 커서로 BB제품2 탭 클릭 → 미매핑 원본 행(Layer 40·50) 선택 → 외부데이터 매핑(2건)** → **적용 버튼을 화면 안으로 스크롤한 뒤 `pressButton`(버튼 눌림 `.tour-pressed` + 커서 리플)으로 클릭을 표현하고** 적용(아래에 BB제품2 2행 추가). 자동채움 결과(BB제품1)와 합쳐 **결과표에 두 제품 모두 반영**. 캡션은 하단 고정 |
| `open-submit` | 상신 확인 모달 오픈 + 특이사항/검토자 샘플값 |
| `submitted` | 모달 닫기 + "요청서가 상신되었습니다" 가짜 성공 토스트 (실제 상신 안 함) |
| `pause` / `resume` | iframe 데모를 **일시정지/재개**. 진행 중 데모를 취소하지 않고 `paused` 플래그만 토글 → pause-aware `sleep`이 그 자리에서 멈췄다가 이어서 진행. 부모는 `paused` 변경 시 자동 전송 |

## 요청서 작성 — phase 흐름 (구현됨)
각 위저드 step 진입 시 **해당 위저드 단계 노드(`.wizard-step[data-step="N"]`)를 강조하며 "다음은 ~~ step입니다" 인트로**를 보여준 뒤 세부 phase로 진행한다. (step1은 인트로 없이 바로 시작)

| step | phase(요지) |
|---|---|
| 1 의뢰 상세 | 위저드 순서(`.wizard-indicator`) → 라인/조합법/제품/조리법(`[data-tour="line-fields"]`) → 필수 *(`.required`) |
| 2 MAP | 인트로 → 목적(`map-reset`,`map-purpose`) → C가문 항목 설명(`[data-tour="map-cfamily"]`, 값 변경 없음) → 지도편차(`map-deviation`) → 예외구역(`map-exception`) → X표시 사진(`map-xmark`) → 가이드(?) 참고(`.guide-badge`) |
| 3 J-ayer | 인트로 → **실제 표 위 커스텀 데모**(cmd `jayer-anim`: 커서+셀선택+Ctrl C/V 칩) → 필터(`[data-tour="jayer-filter"]`) |
| 4 O-ayer | 인트로 → 탭 2개(`oayer-table`,`[data-tour="oayer-tabs"]`) → 정보 탭 TBV/TLV(`oayer-info`,`[data-tour="oayer-info-tbvtlv"]`) |
| 5 뼈찜(BB) | 인트로 → 자동채움 설명(cmd `bb-autofill-open`, 패널만 열기) → 자동채움 적용(cmd `bb-autofill-apply`, 하단 캡션, 커서로 '적용' 눌러 BB제품1 3행) → 매핑(cmd `bb-mapping`, 하단 캡션, BB제품2 탭 클릭 → Layer 40·50 매핑 → **적용 버튼 스크롤+눌림 애니** → BB제품2 2행) → 상신 특이사항+검토자 묶음(`[data-tour="submit-fields"]`) → 완료 알림(cmd `submitted`) |

## 결재 경로 — 독립 단계 (구현됨) · **맨 처음**
가이드의 **첫 단계**는 최종 결재 경로 개관이다. iframe이 아니라 **컴포넌트형 단계**로, `components/ApprovalRouteDiagram.tsx`(props 없는 정적 다이어그램)를 `GuideTourStep.component`로 직접 렌더한다(`제품담당자→검토→RFG→(PHPSI→JOB)∥(OVL[+EUV])→완료`, E는 plel 시·Only MAP은 R까지 조건 주석). 박스는 약어(code) 없이 **이름(label)만** 표시하고, 완료 박스만 `✓ + 완료`를 유지한다. 박스/폰트/간격을 축소(min-width 92·padding 12/14 등)해 '완료'가 다음 줄로 떨어지지 않게 한다. 정적이므로 **챕터 타임라인·일시정지는 숨김**. 결재 경로는 결재현황 페이지(iframe)와 **분리**되어 있으며, 결재현황 iframe 안에는 더 이상 다이어그램을 렌더하지 않는다.

## 결재 현황 — phase 흐름 (구현됨)
`ApprovalPage`에 투어 모드(`?embed=tour`)를 추가했다. 실제 API 대신 샘플 시드(`pages/approvalTourSeed.ts`: 병렬 진행 A / PL 검토 B / 지정 대기 C 3건, MY=A·C)로 목록과 상세를 보여준다. **문서 A는 재상신 이력(history 스냅샷 1건)을 포함**해 상세에서 변경 필드·행이 강조된다. 제목 클릭은 `ApprovalPage` 자체 가짜 커서(`tour-jcursor`)로 연출한다. (결재 경로 다이어그램은 위 별도 단계로 분리되어 이 페이지엔 없다.)

| phase | 요지 |
|---|---|
| 1 소개 | `tour-reset`(모달/필터/커서 초기화) → 필터 탭(`.filter-tabs`) 강조, 결재 현황 페이지 소개 |
| 2 MY 필터 | cmd `my-filter` → `MY` 탭(`[data-tour="approval-my-tab"]`) 강조, 목록이 내 결재(A·C)로 좁혀짐 |
| 3 현재 단계·메일 | 목록 '현재 단계' 컬럼(`[data-tour="approval-stage"]`, 문서 A 행) 강조 — 진행 중 단계·지정 담당자 표시 + 해당 부서/담당자에게 메일 발송 안내 |
| 4 지정하기 | cmd `open-assign` → 문서 C 상세에서 **실제 지정 UI(지정하기→커스텀 드롭다운 펼침→후보 클릭→확인)**로 담당자 배정까지 커서 클릭 시연. 확인 시 `handleAssign`(투어=로컬 반영) → 담당자 배정 토스트. **캡션은 상단(topCaption)** — 하단 footer의 지정 UI를 가리지 않는다 |
| 5 지정 결과 | 상단 캡션 — 확인 시 해당 단계에 담당자가 배정되고 메일이 발송됨을 안내(배정으로 '지정하기' UI는 사라짐) |
| 6 상세 열기 | cmd `open-detail` → **커서가 문서 A 제목을 스크롤·클릭**해 상세 모달 오픈(하단 캡션) |
| 7 상세 탭 안내 | 상세 상단 탭바(`[data-tour="detail-tabs"]`) 강조 — 탭(상세·MAP·JOB·OVL·BB·결재 경로)을 눌러가며 작성 내용을 확인할 수 있다고 안내(클릭 시연 없음) |
| 8 결재 경로 탭 | cmd `page-route` → 상세 **결재 경로 탭**(`[data-tour="approval-route-tab"]`)으로 이동, 팀별·회차별 합의/반려·담당자·일자 이력 안내 (※ 큰 다이어그램과는 다른 회차별 이력 — 그대로 유지) |

| 10 재상신 변경 강조 | 같은 JOB 탭에서 변경 행의 '이력 확인' 버튼(`[data-tour="jayer-hist-btn"]`) 강조 — 재상신 시 변경 항목/행 하이라이트 안내 |
| 11 이력 비교 모달 | cmd `open-rowdiff` → '이력 확인'을 실제 눌러 변경 전/후 비교 모달 오픈(하단 캡션) |

상세 모달은 투어 모드에서 `PagedDetailView`에 `role="MASTER"`를 넘겨 모든 페이지(상세·MAP·JOB·OVL·BB·결재 경로)가 보이도록 한다. 지정하기 시연은 **실제 커스텀 드롭다운 + 확인/취소**를 그대로 사용한다(`data-tour="assign-btn|assign-select|assign-option|assign-confirm"`). J-ayer export 버튼·결재 경로 탭·'이력 확인' 버튼·현재 단계 컬럼에도 `data-tour`를 부여(실제 동작 변경 없음).

시드 문서 A의 병렬 진행 상태: 경로1은 **P 검토중 / J 대기**(P→J 순차이므로 J 단계는 아직 생성 전), 경로2는 **O 검토중**(담당자 지정). 즉 P·J가 동시 검토중으로 보이지 않는다.

전환 효과: phase가 바뀔 때 이전 강조/캡션을 비우고, 다음 항목(스포트라이트·캡션)이 **약 0.8초에 걸쳐 천천히 밝아지며 등장**(`@keyframes guideTourReveal`, 항목 remount로 재생)한다. 흰색 플래시 전환은 제거했다. (강조 대상이 없는 phase는 캡션을 중앙/하단에 표시)

J-ayer 실제표 데모: `jayer-anim` cmd가 `RequestPage`에서 실제 `jayerRows`를 채우고, `useCellSelection.selectCells`로 진짜 셀 선택 하이라이트를 그린다. 가짜 커서/칩은 `tour-jcursor`/`tour-jchip`(position:fixed)로 `data-jtour` 셀 좌표 위에 띄운다.

BB 실제 구동: `bb-autofill`/`bb-mapping`은 `tourRef`(매 렌더 갱신)로 최신 핸들러를 호출해 stale-closure 없이 `handleApplyAutoFill`/`handleApplyMappings`를 실제로 실행한다. 시드: J-ayer Layer `10~50`, 외부데이터 탭1=10/20/30·탭2=40/50. 외부데이터 로드 `useEffect`는 투어 모드에서 `return` 가드 — API 빈 결과가 시드(`makeTourBbExternalData`)를 덮어쓰지 않도록 한다(이 가드가 없으면 외부 데이터 표가 비어 보인다).

## 컴포넌트형 단계 공통 (결재 경로 · 권한관리)
iframe phase 방식 외에 **컴포넌트형 단계**를 지원한다. `GuideTourStep.component`(+ 애니메이션형이면 `animated:true`)를 두면 `GuideTourModal`이 iframe 대신 해당 컴포넌트를 `<step.component embedded paused={paused} />`로 렌더한다.
- **챕터 타임라인은 항상 숨김**(phases 없음).
- **일시정지 버튼**은 `!component || animated`일 때만 노출 → 정적 '결재 경로'는 숨김, 애니메이션형 '권한관리'는 **노출**.
- 미리보기 컨테이너 `.guide-tour-preview-component`(흰 배경+여백).

### 권한관리 (`PermissionUserGroupDemo`, `animated:true`)
- `embedded` prop: 투어 내 임베드 시 자체 lead/callout/`DemoControls`를 숨겨 모달과 자연스럽게 합쳐지게 한다.
- **일시정지**: 데모 공용 훅 `useDemoTimeline(sequence, loopDelayMs, paused)`에 pause 인지 sleep을 추가했다 — `paused`가 true면 진행 중 시퀀스의 `sleep`/`moveTo`/`click`이 그 자리에서 멈췄다가 재개된다. `GuideTourModal`의 일시정지 버튼이 `paused`를 컴포넌트로 전달한다. (다른 데모는 `paused` 미전달 → 기본 false, 영향 없음)
- **⓪ 그룹 인트로(먼저 설명)**: 데모 시작 전 `showIntro` 상태로 무대를 덮는 설명 카드(`.guide-demo-intro`)를 먼저 보여준다. **좌측은 설명 5가지**(같은 역할 묶음 관리 / 그룹 탭은 멤버만 / 알림 메일은 멤버만 / 작성자 임시저장 그룹 내 공유 / 생성 시 본인 자동 포함 — `intro_title`+`intro_b1~b5`), **우측은 멤버 vs 비멤버 비교 미니 애니메이션**(`.guide-demo-cmp`): 멤버 패널에만 그룹 탭→알림 메일→임시저장이 순서대로 페이드인(`@keyframes gdcPop`)되고 비멤버 패널은 비어있음(✕)으로 대비(`cmp_*` 키). 일시정지(`pause`) 시 비교 애니메이션도 `.paused`로 멈춘다. 약 6.5초 후 카드가 사라지고 커서 데모 시작.
- 시연 흐름: (인트로) → 역할 탭(PL) → **+ 사용자 추가** 모달(검색·다중 선택·✕ 제거·추가 → 역할 탭 카운트/목록 반영) → **+ 그룹 만들기** → 그룹명+멤버 선택 → 만들기 → **그룹 탭 생성·이동**.
- **본인 자동 포함**: 백엔드 `group.members.add(request.user)`를 반영해, 생성 결과 그룹 멤버 목록·탭 카운트에 `self_member`('나(본인)')를 포함해 보여준다(모달 단계엔 별도 안내 없음).

(후속 단계) 이력/VOC: `useGuideTourSteps()`에 `path`+`phases`(또는 `component`) 추가.

## i18n (규칙 G — 필수)
프로토타입(`docs/animated_demo_guide_mockup.html`)의 한글 텍스트를 그대로 옮겨선 안 된다. 모달의 모든 노출 텍스트(버튼 라벨, step 제목/설명, 진행 라벨 등)는 **반드시 i18n 키로** 처리한다.

- 위치: `frontend/src/locales/ko.json` / `en.json`
- 키 그룹: 기존 `guide.*`와 충돌하지 않도록 **하위 네임스페이스 `guide.tour.*`** 사용 권장
  - 예: `guide.tour.button`("전체 가이드"), `guide.tour.next`, `guide.tour.prev`, `guide.tour.skip`, `guide.tour.start`("시작하기"), `guide.tour.pause`("일시정지"), `guide.tour.resume`("재생"), `guide.tour.progress`("{{current}}/{{total}}")
  - step별: `guide.tour.steps.{key}.title` / `...description` + phase 캡션 `guide.tour.steps.{key}.flow.*`
- `ko.json` / `en.json`에 **동일 키를 반드시 동시 추가**한다.

## 인터랙션 플로우
1. **idle** — 모달 비활성 상태
2. **열림 (step 0)** — 버튼 클릭 시 "요청서 작성" 단계부터 시작, iframe에 실제 페이지 표시 + 하이라이트 오버레이 자동 진행
3. **다음/이전** — 클릭 시 이전 단계 오버레이 정리 후 다음 단계 iframe/오버레이 시작
4. **일시정지/재생** — 일시정지 클릭 시 오버레이 진행이 멈추고 버튼이 "재생"으로 전환, 재생 클릭 시 이어서 진행. step을 이동하면 일시정지 상태는 해제(기본 재생)된다.
5. **마지막 단계(권한관리)** — "다음" 버튼 텍스트가 "시작하기"로 전환, 클릭 시 모달 종료
6. **건너뛰기** — 어느 단계에서든 즉시 종료

- 모달 내부 구성(위→아래): **우상단 닫기(✕, `.guide-tour-close`)** — **페이지별 탭바(`.guide-tour-tabs`, 클릭 시 해당 단계로 자유 이동)** — iframe 페이지 미리보기 또는 컴포넌트형 단계(+ 하이라이트/커서/캡션 오버레이, 높이 ≤ 52vh) — **동영상형 챕터 타임라인**(iframe phase 단계만; 경과/총 시간 + phase별 챕터 마커, 클릭 시 해당 구간으로 시킹) — 설명 — [일시정지/재생(해당 단계만)] · [이전/다음(마지막=시작하기)]
- 닫기는 **우상단 ✕ 버튼**으로 일원화(기존 하단 "건너뛰기" 버튼 제거). 미리보기 강조 스포트라이트는 대상에 약 14px(`SPOTLIGHT_PAD`) 여백을 둔다. `scrollIntoView`는 세로(`block:'center'`)만 정렬하고 가로(`inline:'nearest'`)는 두어 좌우로 흔들리지 않는다.
- **하이라이트(스포트라이트)는 iframe 미리보기 내부에만 적용**된다 — 실제 앱 화면이 아니라 미리보기 안의 페이지 요소를 강조한다.
- 타임라인 챕터 = phase. 각 챕터 폭은 `phase.hold` 비례, 클릭하면 `seek`로 해당 phase로 이동. 시간은 `hold` 합산 기준의 근사값.

> 영속성 없음: 버튼 클릭 시에만 표시되는 구조이므로 "다시 보지 않기" 및 노출 여부 저장(localStorage/백엔드 필드)은 두지 않는다. 별도 모델/마이그레이션이 필요 없다.

## 접근성 / 예외 처리
- `Esc` 키로 모달 즉시 종료
- 모달 열림 시 포커스 트랩 적용 (Tab으로 모달 밖 요소 이동 불가)
- 권한상 접근 불가한 페이지가 있는 사용자에게는 해당 step을 건너뛰도록 처리
  - 현재 `Navbar.tsx`의 `ROLE_PERMISSIONS` 기준: `NONE` 역할은 홈(`/`)만 접근 가능하므로 사실상 모든 데모 step을 건너뛰어야 함. 나머지 역할(`PL`, `TE_R/P/J/O/E`, `MASTER`)은 5개 페이지 전부 접근 가능하므로 step skip이 필요 없음. → step skip 로직은 `ROLE_PERMISSIONS`와 동일한 기준으로 판단한다.
- 애니메이션이 들어간 데모이므로 `prefers-reduced-motion` 사용자 설정 감지 시 정적 미리보기로 대체하는 것을 고려

## 향후 확장 고려
- 후속 단계(이력/VOC)는 `useGuideTourSteps()`에 `path` + `phases`(iframe 방식) 또는 `component`(자체 데모 컴포넌트)를 추가하면 된다 — iframe 방식 렌더링은 `GuideTourStepPreview`가 공통 처리한다.
- 표/목록 페이지(결재·이력)는 실데이터가 비면 강조 대상이 사라질 수 있으므로, 비었을 때의 대체 캡션 또는 `empty-state` 요소 강조를 고려한다.
- 본 "전체 가이드"는 기존 `/guide` 기능별 가이드(TipTap 콘텐츠 + `guideDemos` 데모)와 별개로 유지한다.
